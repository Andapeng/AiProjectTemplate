# 质量、测试与 AI 评测（Quality & Testing）

## 1. 双轨质量模型

| 轨道 | 关注点 | 典型手段 |
|------|--------|----------|
| **工程测试** | 逻辑正确、回归、契约 | 单元 / 集成 / 契约 / E2E |
| **AI 评测** | 幻觉、遵从格式、稳定性 | Eval 集、提示词回归、人工抽检 |

## 2. 测试金字塔（建议）

```text
        /\
       /E2E\        少量、高价值场景
      /------\
     / 集成/契约 \   中等
    /------------\
   /   单元测试    \ 体量最大、速度快
```

- **契约测试**：与 `specs/contracts/` 对齐；接口变更先改契约与测试再改实现。
- **评测集（Evals）**：存放在 `tests/evals/`，纳入 CI 或定期批跑。

## 3. AI 评测维度（Dimensions）

| 维度 | 说明 | 示例度量 |
|------|------|----------|
| 正确性 | 与参考答案或规则一致 | 精确匹配 / 评分 rubric |
| 遵从性 | 输出结构、工具调用序列 | schema 校验 |
| 稳健性 | 输入扰动下行为 | 同义改写集 |
| 安全 | 注入、越权、泄露 | 红队用例 |
| 成本 | Token、延迟 | 每任务 P95 |

## 4. 基线与回归（Baseline & Regression）

1. 为关键能力建立 **Golden Set**（输入 + 期望属性/参考答案）。
2. 提示词或模型变更时：**同一套评测必须跑**；下降趋势需说明或拒绝合并。
3. 基线版本在 `tests/evals/README.md` 记录（ commit / 标签 ）。

## 5. 发布门禁（Release Gates）

最低门槛（可按项目加减）：

- [ ] **关键契约测试**通过（若项目已启用自动化）
- [ ] **关键 Eval 子集**通过（定义在 `tests/evals/critical/`，或打 `critical` 标签）
- [ ] 无未处理的 **P0/P1** 缺陷（若有缺陷追踪）
- [ ] 安全分级为 L3+ 的数据未被写入日志（见治理文档）
- [ ] **CHANGELOG** 或发布说明已更新（可选：人类发布流程）

## 6. 缺陷与质量度量

建议在 `docs/09-operations/metrics.md`（若存在）跟踪：

- MTTR、变更失败率、评测通过率趋势、每周 Token/费用
